\chapter{Proposed System Architecture}

\section{System Overview}
The proposed system extends the Meadow9+ transcript assistant from a chatbot-centric design into a hybrid retrieval system. Instead of relying on a single interaction mode that faces scalability and latency limitations, the redesigned system supports three complementary capabilities over the same transcript corpus: a summarization pipeline for high-level transcript understanding, a retrieval-augmented generation (RAG) pipeline for conversational question answering, and a keyword-based search interface for direct snippet retrieval.\cite{gao2024retrieval}

To minimise disruption to the existing Meadow9+ infrastructure, the new hybrid retrieval system is deployed as a separate retrieval server that interacts with the current Meadow9+ backend server. This separation reduces the extent of modifications required on the production backend, while enabling the possibility of controlled A/B testing in production to compare the performance of the existing chatbot approach against the new proposed system in this project.

At a high level, the system operates in two distinct phases: an indexing phase and a querying phase.

\section{Indexing Phase}
During the indexing phase, when a user submits a new recording and a corresponding transcription is generated, the transcript continues to be stored in the existing Meadow9+ database as per the current implementation. In addition, the same transcript is forwarded to the Retrieval Server, as illustrated in Figure \ref{fig:system-integration}.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.7\textwidth]{figures/system_integration_transcript_indexing_pipeline.png}
  \caption{System integration: transcript forwarding, summary return, and retrieval indexing}
  \label{fig:system-integration}
\end{figure}

The Retrieval Server performs two tasks in parallel. First, it generates a raw transcript summary and returns it to the Meadow9+ Server. This summary provides high-level abstractions of the transcript, including overview, topic, outline, segment, action, and entity-level information. The Meadow9+ Server can then store this returned summary in the database and return it to the user as a high-level summary of the transcript.

Second, the Retrieval Server performs the processing, chunking, and embedding stages required to prepare the transcript for later retrieval. Figure \ref{fig:indexing-pipeline} shows this high-level indexing pipeline, where the raw transcript is processed and transformed into indexed representations that can later support retrieval-augmented question answering and search.

\section{Hybrid Retrieval System}
This architecture supports a Hybrid Retrieval System consisting of three complementary components. The first component is the Summarization Pipeline, which provides users with high-level transcript understanding by generating transcript summaries from newly uploaded transcripts. This pipeline is also crucial because the produced summary works hand-in-hand with the RAG pipeline to support more vague, high-level questions that benefit from transcript-level framing rather than only fine-grained evidence retrieval. The second component is the RAG Pipeline, which supports conversational question answering by retrieving relevant transcript evidence and grounding LLM responses in that retrieved context. The third component is Keyword-Based Search, which enables faster direct retrieval for keyword-based, date-based, and speaker-based lookup tasks.

Together, these three components enable the system to support both high-level understanding and fine-grained retrieval over long transcripts. The detailed implementation of these components will be discussed in Chapter 4.

\section{Evaluation Architecture}
The current evaluation architecture focuses on the conversational QA path of the RAG subsystem. At a high level, the system evaluates retrieval and answer generation using a dedicated question--answer dataset derived from transcript data, then runs batch evaluation to measure how effectively the system retrieves relevant evidence and produces grounded answers.

\begin{figure}[H]
  \centering
  \includegraphics[width=1.0\textwidth]{figures/evaluation_pipeline_retrieval_and_generation_metrics.png}
  \caption{Evaluation pipeline for the RAG question-answering subsystem}
  \label{fig:evaluation-pipeline}
\end{figure}

The evaluation outputs include retrieval metrics, generation metrics, and speed benchmarking. Retrieval quality is assessed using ranking-based measures such as hit rate and mean reciprocal rank (MRR), while generation quality is assessed using lexical and semantic correctness measures.\cite{wikipedia2024mrr,vanrijsbergen1979ir} In addition, the system benchmarks the RAG-based QA subsystem against a pure-LLM baseline to compare latency and overall response efficiency across different transcript lengths, as shown in Figure \ref{fig:speed-benchmark}.

\begin{figure}[H]
  \centering
  \includegraphics[width=1.0\textwidth]{figures/benchmarking_rag_vs_pure_llm_comparison.png}
  \caption{Benchmarking: RAG-based QA subsystem vs. pure LLM baseline}
  \label{fig:speed-benchmark}
\end{figure}

This evaluation architecture focuses on the RAG pipeline, as this is the subsystem that can be directly compared against the existing chatbot in terms of performance, scalability, and latency. The summarization pipeline and keyword-based search interface are presented as implemented capabilities of the proposed system, but they are not the primary focus of the comparative evaluation.

\chapter{System Implementation}
This chapter presents the implementation of the Hybrid Retrieval System. The final system comprises three coordinated subsystems built on shared transcript ingestion and retrieval infrastructure: a summarization pipeline for high-level understanding, a RAG pipeline for grounded question answering, and a keyword-based search workflow for direct snippet retrieval.

\section{Implementation Overview}
\subsection{Shared Processing Backbone}
The final implementation was designed around one shared transcript-processing backbone rather than three completely independent systems. Each uploaded transcript is parsed once, enriched with timestamps and speaker metadata, and stored in a global retrieval layer. On top of this backbone, the system produces both fine-grained transcript chunks and summary-derived documents, allowing the same underlying corpus to support detailed QA, broad topic-level retrieval, and search-oriented lookup.

This architecture reduced duplicated infrastructure and made the three subsystems mutually reinforcing. In particular, the summary pipeline does not only generate a user-facing summary; it also produces retrieval artefacts that improve broad-query handling in the chatbot and transcript ranking in the search bar.

\subsection{End-to-End Processing Flow}
The implementation flow follows a consistent progression from raw transcript to user-facing capability. First, incoming SRT transcripts are parsed into structured segments with timestamps and speaker annotations. Second, those segments are merged into retrieval-sized chunks and transformed into document objects for embedding and storage. Third, the same transcript is passed through a summarization path that produces one canonical summary and multiple summary-derived retrieval documents. Finally, the chatbot and search bar query the shared retrieval layer with different ranking and output strategies. This shared flow is the main reason the final system can support three different interaction styles without maintaining three separate backends.

\subsection{Global Storage Design}
All transcript chunks are stored in a single global index rather than separate transcript-specific indices. This simplifies deployment, supports cross-transcript retrieval, and allows transcript scoping through metadata filters. The same storage layer also contains summary-derived documents, so broad transcript-level evidence and low-level chunk evidence can be retrieved through one consistent interface.\cite{langchain_chroma,chroma_persistence_sqlite}

\subsection{Shared Metadata Model}
The same metadata model is reused across subsystems. Transcript identifiers, timestamps, speaker information, and upload-date fields are attached during ingestion and then reused for transcript-scoped retrieval, user-facing citations, summary-document generation, and keyword-based filtering. This was an important implementation choice because it allowed search, summarization, and QA to operate over one coherent document representation instead of maintaining separate data-preparation logic for each feature.

\subsection{Role-Specific Model Stack}
The model stack is also separated by role rather than handled by one generic LLM endpoint. Embeddings are generated by a dedicated locally-hosted jina-embeddings-v3-f16 model because retrieval quality depends on stable dense representations rather than generative behavior. Reranking is handled by a separate cross-encoder bge-reranker-v2-m3-FP16 model because pairwise relevance scoring has different accuracy and latency requirements from both embeddings and answer generation. The local Qwen3-14B-Q4 model is then reused for the two generative tasks in the system, namely summarization and grounded answer generation. For evaluation pipelines, DeepSeek, via the DeepSeek API, is used to generate ground-truth data. 

All local models are hosted using llama.cpp server, which provides efficient inference with optimized CPU/GPU acceleration and OpenAI-compatible API endpoints.\cite{jina_embeddings_v3,bge_reranker_v2_m3,qwen3_14b_base,deepseek_api_docs,llama_cpp_server} These models are hosted as separate services to better simulate a production deployment environment, where different inference components may be scaled, tuned, or monitored independently. This separation also makes model replacement easier, because any one component can be swapped out when a better embedding model, reranker, or generator becomes available without requiring the entire pipeline to be redesigned.

\section{Summarization Pipeline Implementation}
The summarization pipeline forms the first subsystem of the Hybrid Retrieval System. It was added to support high-level transcript understanding and to provide transcript-level retrieval artefacts for broad or vague downstream queries. A single full-context LLM call is not reliable for long transcripts, so the implementation adopts a hierarchical multi-stage design in which summarization is decomposed into chunking, per-chunk extraction, and global synthesis.

\subsection{Transcript Chunking for Summarization}
The summarization path first splits the transcript into token-counted chunks rather than relying on a simple character limit. Chunk boundaries are aligned as far as possible with speaker turns so that local summaries preserve conversational flow, participant structure, and topic boundaries. Small overlaps are carried across adjacent chunks to reduce information loss at chunk boundaries, which is particularly important for discussions that span multiple turns or contain deferred references.

This chunking strategy differs in purpose from transcript chunking for retrieval. In the summarization subsystem, chunking exists to keep each extraction request within the local model's practical input budget while preserving enough local coherence for reliable structured extraction.

\subsection{Per-Chunk Structured Extraction}
Each chunk is then processed asynchronously by an LLM to generate a structured JSON-style chunk summary. Each chunk summary captures local discussion points, entities, and action items, forming an intermediate representation that is easier to merge than raw transcript text.

\subsection{Hierarchical Merge and Canonical Summary Schema}
After chunk extraction, the intermediate summaries are merged into a canonical transcript-level summary.

The resulting canonical summary is stored in structured JSON form. Its fields capture transcript-level framing information such as participants, conversation type, purpose, overview, key topics, detailed topics, recurring themes, outline sections, notable segments, action items, and entities. This schema is richer than a simple free-text abstract and is designed to support both user-facing summarization and downstream retrieval over specific summary sections.

\subsection{Summary Embedding Documents}
The summarization subsystem does not stop at producing one canonical summary JSON file. During ingestion, the system also transforms the summary into multiple semantically focused retrieval documents, such as overview, topic, outline, segment, action, entity, and fallback miscellaneous documents. Each of these derived documents is tagged with summary-specific metadata and embedded into the same retrieval infrastructure used for transcript chunks.

This design solves two problems simultaneously. First, it avoids representing the entire summary as one monolithic vector, which would be too coarse for topic-level matching. Second, it allows broad or transcript-level queries to retrieve focused summary sections instead of only low-level transcript snippets. These summary documents therefore function as retrieval artefacts, not merely display data.

\section{RAG Pipeline Implementation}
The RAG pipeline remains the primary subsystem for conversational question answering. Its role is to transform a user question into a grounded answer by retrieving relevant evidence, assembling context, and invoking the LLM under explicit grounding constraints.\cite{gao2024retrieval}

\subsection{Transcript Chunking and Shared Indexing}
At ingestion time, transcripts are parsed from SRT into structured segments, enriched with timestamps and speaker metadata, and merged into retrieval-sized chunks. These chunks are embedded and stored in the same global retrieval layer described earlier in the chapter, which allows the QA subsystem to reuse shared metadata filtering, transcript scoping, and summary-derived retrieval artefacts rather than maintaining a separate storage path. This shared representation was important in the final system because it kept conversational retrieval aligned with the same underlying corpus used by summarization and keyword search. Detailed implementation mechanics for transcript parsing, token-based chunk merging, document construction, and vector-store persistence are provided in Appendix A.

\subsection{Hybrid Retrieval and Ranking}
\subsubsection{Complementary Retrieval Signals}
Retrieval itself is implemented as a hybrid process rather than a single semantic search call. As illustrated in Figure \ref{fig:rag-hybrid-overview}, dense vector retrieval provides robustness to paraphrasing and conceptual similarity, BM25 preserves exact lexical matching for names and technical terms, phonetic retrieval improves recall when ASR introduces sound-alike spelling distortions, and fuzzy matching captures minor character-level corruption in rare tokens.\cite{robertson2009bm25,black2021doublemetaphone,levenshtein1965,jina_embeddings_v3,metzger2024measuring} These signals are intentionally complementary because transcript corpora contain both semantic variation and transcription noise, making any one method insufficient across all query types.

\subsubsection{Role of Individual Retrieval Methods}
Each retrieval signal addresses a different failure mode. Vector retrieval is strongest when the user paraphrases content that does not appear verbatim in the transcript. BM25 is most reliable when the query contains explicit lexical anchors such as product names, brands, or technical terms. Phonetic retrieval is valuable when automatic speech recognition preserves pronunciation but distorts spelling, while fuzzy matching helps recover rare terms affected by small character-level corruption. In combination, these methods produce a broader and more robust candidate set than any single retriever could provide on its own.

\subsubsection{Fusion and Precision-Oriented Reranking}
The outputs of these retrievers are not passed directly to generation. Instead, the system first combines them with Reciprocal Rank Fusion to reward candidates that are consistently relevant across methods, and then applies a cross-encoder reranker to the fused candidate pool for precision-oriented final ordering.\cite{cormack2009reciprocal,nogueira2019passage} This two-stage ranking strategy balances recall and precision: the first stage casts a broad net over noisy transcript data, while the second sharpens the evidence set that is ultimately shown to the language model. The detailed implementation of the individual retrievers, rank fusion, and reranker response handling is deferred to Appendix A.

\subsection{Context Assembly and Prompt Construction}
\subsubsection{Grounded Context Formatting}
After reranking, the highest-ranked chunks are converted into grounded context blocks that preserve transcript identifiers, timestamps, and citation metadata. These blocks form the evidence layer of the final prompt, ensuring that answer generation remains tied to inspectable transcript passages rather than unsupported model recall.\cite{langchain_prompts,langchain_chatprompt,langchain_messages_placeholder}

\subsubsection{Intent-Aware Route Selection}
Prompt assembly itself is intent-aware. Before the final prompt is built, a lightweight rule-based router classifies the query and selects a route that determines how much context to retrieve and how that context should be packed. This routing step affects retrieval depth, neighbour expansion, transcript-summary use, transcript-anchor injection, and the balance between transcript-level framing and local evidence; the detailed implementation of this classifier and route-aware prompt construction is provided in Appendix A.

The broader routes, \texttt{summary\_path} and low-confidence \texttt{hybrid\_path}, combine transcript-level framing with supporting chunks so the model can answer vague or transcript-wide questions with better coverage. By contrast, \texttt{fine\_grained\_path} stays local and detail-oriented for precise factual questions, while \texttt{topic\_path} occupies a middle ground: it has its own retrieval profile but still uses standard retrieved-chunk assembly rather than the fuller frame-aware packing used by the broader routes. This design allows the system to expand context when the query is broad, while retaining precision when the user is asking for a specific detail.

\subsubsection{Modular Prompt Pipeline}
Overall, the RAG pipeline was kept modular rather than collapsed into one opaque end-to-end model call. Chunk construction, hybrid retrieval, reranking, and prompt assembly can therefore be tuned independently as transcript characteristics or product requirements change. Detailed implementation mechanics, data structures, and supporting screenshots are deferred to Appendix A.

\section{Keyword-Based Search Implementation}
The keyword-based search subsystem complements the chatbot by prioritising direct retrieval rather than answer generation. It is intended for users who want inspectable search results quickly, such as all mentions of a product, statements by a specific speaker, or snippets within a particular date range.

\subsection{Keyword-First Retrieval Design}
The keyword-based search subsystem reuses the same retrieval primitives as the RAG pipeline, namely vector search, BM25, phonetic search, fuzzy matching, Reciprocal Rank Fusion, and reranking. Its orchestration, however, is designed for global search across the transcript collection rather than transcript-scoped conversational QA. In this interaction mode, BM25 is weighted more heavily so that exact lexical matches dominate for queries involving names, brands, technical terms, and other explicit keywords.

This design also changes the presentation granularity of the results. The chatbot operates within a selected transcript and retrieves a flat set of evidence chunks to support answer generation, whereas the search bar performs global retrieval and returns results grouped by transcript. Grouping at transcript level is more appropriate for search-oriented interaction because it allows users to identify the most relevant transcripts first before examining the supporting snippets within each result group.

\subsection{Two-Phase Search Flow}
The search subsystem performs retrieval in two phases. In Phase 1, the system ranks transcripts using summary-derived documents only. This provides a transcript-level relevance signal that is cleaner than mixing all low-level content chunks together. Summary chunks are especially useful here because they represent the high-level contents of a transcript and can therefore act as transcript proxies.

In Phase 2, the selected transcripts are searched again at chunk level. Retrieval is scoped to one transcript at a time, the fused results are reranked, and the final top chunks are returned as grouped results.

\subsection{Date and Speaker Filtering}
The search bar supports natural-language date filters and speaker filters in addition to free-text queries. Date expressions are parsed into time ranges using the dateparser library, then matched against transcript upload metadata to ensure all relevant transcripts are considered before chunk-level retrieval.\cite{dateparser_docs} Speaker filtering relies on speaker metadata extracted during transcript parsing and mapped onto the corresponding chunk metadata, enabling transcript-scoped filtering by one or more speakers.

\subsection{Relationship to the RAG Subsystem}
Although the search bar and chatbot share the same retrieval building blocks, they are optimised for different interaction styles. The RAG subsystem uses retrieval to assemble grounded context for answer generation, whereas the search subsystem uses retrieval to return directly inspectable snippet lists. This separation avoids overloading the chatbot interface with direct lookup tasks while still allowing both subsystems to benefit from the same shared indices and metadata.

\section{Deployment Implementation}
The final step in the project was to expose these capabilities through a deployable retrieval server so that the implemented pipelines could be used by the broader Meadow9+ system rather than remaining as development-only components. The core deployment architecture has already been implemented, and most operational interfaces required by the application are already in place. Although the system has not yet been rolled out as a full production deployment, moving from the current implementation to production mainly requires environment-specific integration and infrastructure changes rather than a redesign of the retrieval system itself.

\subsection{Retrieval Server Deployment Architecture}
The implemented retrieval system is packaged as a separate FastAPI-based retrieval server, consistent with the architectural direction introduced in Chapter 3. This deployment decision keeps the new retrieval logic isolated from the existing Meadow9+ backend while still making the system accessible through a stable service interface. In practice, the runtime is composed of four cooperating services: a FastAPI application layer, a generation service for summarization and answer synthesis, an embedding service for dense retrieval, and a reranking service for precision-oriented evidence ordering. This separation preserves the modular implementation described earlier in the chapter while making the runtime easier to maintain, tune, and extend.\cite{fastapi_docs,pydantic_docs,llama_cpp_server}

The service-oriented deployment also reflects the role-specific model design introduced in Section 4.2.4. Embedding, reranking, and generation are not treated as one monolithic inference endpoint; instead, each is exposed as an independent runtime component. This allows individual services to be replaced, reconfigured, or scaled without redesigning the rest of the retrieval pipeline, which is important for a system expected to evolve as better local models become available. In this sense, the current retrieval server should be understood as an integration-ready service layer for Meadow9+ rather than as a claim that the final production rollout has already been completed.

\subsection{Persistent Runtime Storage}
The deployed server uses persistent storage with a clear separation of responsibilities. Structured transcript metadata, raw transcript text, upload timestamps, and canonical summary outputs are currently stored in SQLite so that uploaded transcripts remain accessible across server restarts and can be listed, retrieved, and filtered through the application layer. In the present implementation, SQLite functions primarily as a lightweight placeholder for the MongoDB-based transcript storage already used by the existing Meadow9+ server, allowing the retrieval workflow to be implemented and tested locally without depending on direct production-database integration. In parallel, transcript chunks and summary-derived retrieval documents are embedded and stored in a persistent Chroma vector store, allowing the same deployed corpus to support semantic retrieval, summary-level transcript ranking, and transcript-scoped chunk search.\cite{langchain_chroma,chroma_persistence_sqlite}

This storage design is important because it avoids treating retrieval as an in-memory prototype while still remaining compatible with the broader Meadow9+ architecture. Since SQLite is only simulating the existing MongoDB-backed transcript store, integration into the production environment mainly requires adapting the persistence layer to the server's existing database workflow, while leaving the downstream retrieval pipeline unchanged.

\subsection{End-to-End Deployed Processing Flow}
When a new transcript is uploaded, the deployed server validates the input, parses the transcript into structured segments, constructs retrieval-sized chunks, generates the transcript summary, stores transcript artefacts in persistent storage, and then indexes both chunk documents and summary-derived documents into the shared vector store. As a result, ingestion, summarization, persistence, and indexing are all handled within one deployed runtime path rather than through disconnected development scripts.

In the current implementation, this workflow is triggered via transcript-upload endpoints, enabling the full pipeline to be tested and evaluated independently during development. For production deployment, the same ingestion pipeline can be seamlessly adapted to receive transcript data directly from the Meadow9+ backend server instead of manual uploads. Under this integration, only the source of the transcript data changes, while all downstream processing stages—namely parsing, chunking, summarization, persistence, indexing, and retrieval—remain unchanged.

Once transcripts have been ingested, the same deployment supports the system's three user-facing interaction modes. The backend can list uploaded transcripts, return stored summary or raw-content views for a selected transcript, execute keyword-based global search over the uploaded corpus, and serve transcript-aware chatbot or one-off query requests through the RAG pipeline. This end-to-end deployment is therefore not limited to inference alone; it operationalizes the entire transcript lifecycle from upload to retrieval-driven interaction.

\subsection{Streaming Chat Responses}
The deployed retrieval server also supports streaming chat responses, which was missing from the Meadow9+ chatbot implementation. Instead of waiting for the full answer to be generated before returning any output, the server can stream partial answer tokens to the client as they are produced. This improves perceived latency because users receive visible progress early rather than experiencing one long blocking delay.

\subsection{Retrieved Evidence in Chat Responses}
The deployed chat interface also returns the retrieved transcript chunks used to support answer generation. Rather than exposing only a final generated response, the system can surface supporting transcript excerpts together with metadata such as timestamps and source identifiers. This design directly reflects the transparency goals discussed in Chapter 2, where Meadow9+ was motivated as a system that should expose inspectable evidence rather than behave as a fully opaque chatbot.

Returning retrieved chunks improves verifiability because users can inspect which transcript segments were actually used to ground an answer. This in turn improves user trust and confidence, especially in professional settings where incorrect attribution or hallucinated conclusions may have practical consequences.

\subsection{Operational Interfaces and Health Monitoring}
The FastAPI layer provides the operational boundary between the retrieval server and the rest of the system. Instead of exposing internal LangChain pipelines directly, the implementation wraps transcript ingestion, transcript retrieval, search, question-answering, streaming chat, and health-monitoring capabilities behind explicit service interfaces. This makes the deployed system easier to integrate with the Meadow9+ backend and keeps application-facing behavior separate from the lower-level implementation details of chunking, retrieval, and prompting.\cite{fastapi_docs,pydantic_docs} A compact summary of the implemented interface categories is provided in Appendix A, Table \ref{tab:retrieval-server-interfaces}.

Operational monitoring is also built into the deployment. A health-check interface reports whether the retrieval runtime and its upstream dependencies are available, including the embedding, reranking, and generation services, as well as whether the retrieval layer has been initialized successfully. This is a practical deployment requirement because the final system depends on multiple cooperating components rather than one standalone process. Detecting service availability at runtime makes failures easier to diagnose and provides a clearer integration contract for the surrounding application stack.

Overall, the implemented deployment transforms the Hybrid Retrieval System from a set of individual pipelines into a usable backend service. The final system is therefore not only a prototype of summarization, RAG, and search techniques, but an operational retrieval server implementation whose core interfaces and processing paths are already in place, making integration into Meadow9+ achievable with relatively small deployment-specific changes rather than a fundamental redesign.

\section{Prototype Flutter Client}
In addition to the backend retrieval server, a prototype Flutter client was also implemented to exercise the new retrieval capabilities through a mobile-oriented user interface. This frontend work is not a primary contribution of the project and remains technically outside the formal scope of the retrieval-system implementation. It is nevertheless useful to mention briefly because it demonstrates that the deployed backend interfaces can already support an end-to-end application workflow.

The prototype client exposes the three main user-facing interaction modes introduced in this project. The summary view presents the generated meeting summary in a structured form rather than as one block of plain text, making the output easier to scan and navigate. The Ask AI view surfaces the number of retrieved sources used to answer a question and allows users to expand those supporting chunks, directly operationalizing the transparency and confidence goals discussed earlier in this chapter. The global search view integrates the keyword-based transcript search workflow, showing that the same backend can also support direct retrieval-oriented interaction outside the chatbot interface.

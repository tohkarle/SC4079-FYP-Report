\chapter{Evaluation and Results}
This chapter presents the evaluation pipeline implementation, experimental results, and discussion used to assess the retriever system and its end-to-end performance.

\section{Evaluation Pipeline Implementation}
A rigorous evaluation framework was developed to assess both the quality and performance of the RAG system. The evaluation pipeline comprises dataset generation, metric computation, and comparative benchmarking components.

This evaluation framework is designed to operate automatically, eliminating the need for manual human intervention. Traditional evaluation approaches require extensive manual effort, including human annotation of ground truth question-answer pairs, manual inspection of retrieved document relevance, and subjective assessment of generated answer quality \cite{ares2023}. Such manual processes are time-consuming, costly, and difficult to scale across large transcript corpora.

With the advancement of LLM, it is now feasible to design an automated evaluation framework that leverages LLM capabilities for both dataset generation and answer quality assessment \cite{ragas2023}. The proposed pipeline automates ground truth generation using an LLM to synthesize contextually relevant question-answer pairs from transcript excerpts (Section 5.1.1), computes retrieval metrics algorithmically through temporal overlap detection (Section 5.1.2), and employs an LLM judge to evaluate semantic correctness of generated answers (Section 5.1.3). This automation enables rapid iteration during system development, facilitates reproducible evaluation across configuration changes, and scales efficiently to accommodate additional transcripts without proportional increases in human labor.

\subsection{Dataset Generation Methodology}

Ground truth evaluation datasets were generated automatically using an LLM-based approach. The generation process follows a structured pipeline that transforms raw SRT transcripts into question-answer pairs with temporal metadata for retrieval evaluation.

\textbf{5.1.1.1 Context Window Grouping}

The grouping algorithm aggregates small SRT chunks into larger context windows suitable for LLM processing. The grouping process employs a greedy first-fit algorithm that iterates through SRT chunks sequentially, accumulating text until a maximum character threshold is reached.

The algorithm maintains temporal boundaries by tracking the start and end timestamps of each grouped window, ensuring that generated questions can be linked back to specific time ranges in the source transcript. This temporal metadata is critical for subsequent retrieval evaluation.

The prompt template enforces three critical requirements. First, questions must be self-contained, avoiding pronouns like "he," "she," or "it" without clear antecedents. This ensures questions can be answered without additional context beyond the retrieved chunks. Second, answers must be directly supported by the provided text, preventing hallucination. Third, the output must be valid JSON, enabling automated parsing and validation.

The template requests 2-3 question-answer pairs per context window, balancing dataset size with question quality. Requesting more pairs per window (e.g., 5-7) resulted in lower-quality questions that repeated similar information.

Each generated QA pair is enriched with three metadata fields. The source\_start and source\_end fields capture the temporal boundaries of the context window from which the question was generated. This temporal metadata enables retrieval relevance checking during evaluation by allowing the system to verify whether retrieved chunks overlap with the ground truth time range. The context field stores the full context window text, enabling manual validation of question quality and answer correctness.

\subsection{Retrieval Metrics}

Retrieval metrics evaluate the system's ability to find relevant chunks from the transcript before answer generation. Two complementary metrics are used: Hit Rate (recall-oriented) and Mean Reciprocal Rank (ranking-oriented).

\textbf{5.1.2.1 Hit Rate Computation}

Definition: Proportion of queries where at least one retrieved chunk has temporal overlap with the ground truth timestamp range \cite{vanrijsbergen1979ir}. The formula is shown in Figure \ref{fig:fig51}:

\begin{figure}[H]
  \centering
  \includegraphics[width=1.0\textwidth]{figures/hit_rate_formula_for_retrieval_evaluation.png}
  \caption{Hit Rate formula for retrieval evaluation}
  \label{fig:fig51}
\end{figure}

Hit Rate is computed through temporal overlap detection between retrieved chunks and ground truth time ranges. Figure \ref{fig:fig73} shows the temporal overlap detection algorithm:

\begin{figure}[H]
  \centering
  \includegraphics[width=1.0\textwidth]{figures/interval_intersection_algorithm_for_retrieval_relevance_checking.png}
  \caption{Interval intersection algorithm for retrieval relevance checking}
  \label{fig:fig73}
\end{figure}

The algorithm implements standard interval intersection logic: two intervals [a,b] and [c,d] overlap if and only if a < d AND b > c. For example, if a QA pair was generated from context spanning [10s, 20s] and a retrieved chunk covers [15s, 25s], the intervals overlap and the chunk is considered relevant. The function returns both a boolean hit flag and the 1-indexed rank of the first relevant chunk, enabling simultaneous Hit Rate and MRR computation.

\textbf{5.1.2.2 Mean Reciprocal Rank (MRR)}

Definition: Average of reciprocal ranks of the first relevant result, as illustrated in Figure \ref{fig:fig52} \cite{wikipedia2024mrr}:

\begin{figure}[H]
  \centering
  \includegraphics[width=1.0\textwidth]{figures/mean_reciprocal_rank_mrr_formula_and_definition.png}
  \caption{Mean Reciprocal Rank (MRR) formula and definition}
  \label{fig:fig52}
\end{figure}

If a hit is detected, the MRR contribution is computed as 1.0 divided by the rank. An MRR of 1.0 indicates perfect retrieval performance, where the first relevant result is consistently ranked at position 1. An MRR of 0.5 implies that the first relevant result appears at rank 2 on average, while an MRR of 0.333 suggests it appears at rank 3 on average. Conversely, an MRR of 0.0 indicates retrieval failure, where no relevant results are returned for the evaluated queries.

MRR provides a more nuanced assessment than Hit Rate by penalizing lower-ranked relevant results. A system with Hit Rate = 1.0 and MRR = 0.5 successfully retrieves relevant chunks for all queries but often places them at rank 2 or lower, indicating suboptimal ranking quality.

\subsection{Generation Metrics}

Generation metrics evaluate the quality of answers produced by the LLM after retrieval. Two complementary metrics are used: F1 Score (lexical overlap) and LLM Judge Accuracy (semantic correctness).

\textbf{5.1.3.1 F1 Score Calculation}

Definition: Harmonic mean of precision and recall based on token-level overlap between generated and ground truth answers, as shown in Figure \ref{fig:fig53} \cite{vanrijsbergen1979ir}:

\begin{figure}[H]
  \centering
  \includegraphics[width=1.0\textwidth]{figures/f1_score_calculation_with_precision_and_recall.png}
  \caption{F1 score calculation with precision and recall}
  \label{fig:fig53}
\end{figure}

F1 Score computation requires token-level comparison between generated and ground truth answers. Figure \ref{fig:fig76} shows the normalization process:

\begin{figure}[H]
  \centering
  \includegraphics[width=1.0\textwidth]{figures/text_preprocessing_for_token_level_comparison.png}
  \caption{Text preprocessing for token-level comparison}
  \label{fig:fig76}
\end{figure}

To enable robust token-level comparison, a lightweight normalization step is applied to both texts prior to metric computation. This process converts all characters to lowercase, removes punctuation and other non-word symbols via regex, and collapses repeated whitespace. This focuses the evaluation on lexical content rather than formatting differences.

Figure \ref{fig:fig77} shows the F1 computation using Counter intersection:

\begin{figure}[H]
  \centering
  \includegraphics[width=1.0\textwidth]{figures/precision_recall_and_harmonic_mean_calculation_with_counter_intersecti.png}
  \caption{Precision, recall, and harmonic mean calculation with Counter intersection}
  \label{fig:fig77}
\end{figure}

The Counter intersection operation returns the minimum count for each token present in both prediction and ground truth. For example, if the prediction contains ["apple", "apple", "banana"] and the ground truth contains ["apple", "banana", "banana"], the intersection yields \{"apple": 1, "banana": 1\}. The num\_same variable sums these counts to get the total number of overlapping tokens.

Precision measures what fraction of the predicted tokens are correct. Recall measures what fraction of the ground truth tokens were predicted. F1 is the harmonic mean of precision and recall, providing a balanced measure that penalizes both over-generation (low precision) and under-generation (low recall).

The F1 score provides a balanced measure of lexical overlap between prediction and ground truth, but may underestimate semantic correctness when valid paraphrasing or synonym substitution is present. For example, "quarreled" vs "argued" would reduce F1 despite semantic equivalence.

\textbf{5.1.3.2 LLM Judge Accuracy (End-to-End Accuracy)}

Definition: Binary semantic correctness evaluated by an LLM judge, assessing whether the generated answer is semantically consistent with the ground truth \cite{liu2023geval}. Figure \ref{fig:fig54} shows the prompt template:

\begin{figure}[H]
  \centering
  \includegraphics[width=1.0\textwidth]{figures/llm_judge_prompt_template_for_semantic_correctness_evaluation.png}
  \caption{LLM judge prompt template for semantic correctness evaluation}
  \label{fig:fig54}
\end{figure}

The LLM judge receives a formatted prompt containing the question, ground truth answer, and generated prediction. The prompt explicitly instructs the judge to output only "CORRECT" or "INCORRECT" based on semantic consistency. The response is parsed using a hierarchical decision logic: if "CORRECT" appears without "INCORRECT", the answer is marked correct; if "INCORRECT" appears, it is marked incorrect; otherwise, a fallback heuristic checks for "CORRECT" in the content. This handles edge cases where the judge provides additional explanation alongside the verdict.

The judge LLM uses temperature 0.0 for deterministic evaluation, reducing variability in scoring. Error handling returns False on judge failure, taking a conservative approach that prefers false negatives to false positives.

Both F1 and LLM Judge metrics are computed for each generated answer and accumulated for averaging. This dual evaluation provides complementary perspectives: F1 captures lexical fidelity while LLM Judge captures semantic correctness.

\subsection{Separation of Retrieval and Generation Evaluation}

\textbf{Computed Before Generation:} Hit Rate and MRR. These metrics do not depend on the generated answer and evaluate only the retrieval pipeline's ability to find relevant chunks.

\subsection{Speed Evaluation Pipeline}

This section introduces a lightweight speed evaluation pipeline that is separated from the main evaluation pipeline to enable an unbiased latency comparison between the RAG-based approach and a Pure LLM baseline. The pipeline reuses the same ground truth dataset described in Section 5.1.1, including identical questions, reference answers, and temporal metadata, to ensure consistency across evaluations. This separation is necessary due to prompt caching effects in modern LLM services, which can artificially distort latency measurements under sequential execution \cite{openai_prompt_caching}. The mitigation strategy adopted to address this issue is discussed in detail in Section 5.1.5.1.

\textbf{5.1.5.1 Interleaved Execution Design}

The speed evaluation pipeline addresses prompt caching bias through interleaved execution.

LLM services cache recently seen prompts to improve efficiency \cite{openai_prompt_caching}. A sequential benchmarking setup could unintentionally allow the Pure LLM baseline to benefit from warmed caches, thereby underestimating its true latency and producing unfair comparisons. To address this, the pipeline alternates between RAG and Pure LLM execution on a per-question basis, ensuring both methods experience similar caching conditions and enabling a more reliable and unbiased measurement of relative runtime performance, as illustrated in Figure \ref{fig:fig60}.

\begin{figure}[H]
  \centering
  \includegraphics[width=1.0\textwidth]{figures/interleaved_execution_pattern_for_unbiased_speed_benchmarking.png}
  \caption{Interleaved execution pattern for unbiased speed benchmarking}
  \label{fig:fig60}
\end{figure}

\textbf{5.1.5.2 Warmup Phase}

The evaluation pipeline includes a warmup phase, which is designed to reduce cold-start effects and improve the reliability of latency benchmarking. Specifically, it triggers just-in-time (JIT) compilation and warms up the Python runtime and dependent libraries, initializes and primes the LLM, embedding, and reranker services, and preloads model weights into GPU memory to avoid first-run loading overhead. In addition, it establishes and stabilizes HTTP connections to mitigate transient network latency and connection setup costs during the measured runs.

Warmup results are excluded from timing statistics to measure steady-state performance.

RAG provides substantial speed improvements (1.4x to 7x) with either comparable or slightly reduced quality. For small transcripts, RAG is strictly superior (faster and equal/better quality). For large transcripts, RAG offers a favorable trade-off with massive speedup for small accuracy reduction.
